## Q1) Redis가 빠른 이유

### 내 답변
인메모리 DB라서 SSD/HDD + 네트워크 I/O가 아닌 RAM을 사용하고, 싱글 스레드 기반이라 블로킹이 없음

### 모범 답변
**1. 인메모리 저장**
- 데이터를 디스크가 아닌 RAM에 저장 → 디스크 I/O(SSD ~0.1ms, HDD ~10ms) 대비 RAM 접근(~100ns)은 수십~수백 배 빠름
- 네트워크 I/O 병목도 없음

**2. 싱글 스레드 이벤트 루프**
- 단일 스레드로 명령어를 순차 처리 → Lock/Context Switch 오버헤드 없음
- I/O는 멀티플렉싱(epoll/kqueue)으로 비동기 처리하므로 블로킹 없이 높은 처리량 유지

**3. 단순한 자료구조**
- String, List, Hash, Set, ZSet 등 목적에 맞는 자료구조 제공 → 연산 시간복잡도가 낮음

**추가 언급 포인트:**
- 싱글 스레드이므로 하나의 무거운 명령(KEYS *, FLUSHALL 등)이 전체를 블로킹할 수 있음 → 운영 시 주의 필요
- Redis 6.0부터 네트워크 I/O는 멀티 스레드로 처리 (명령 실행은 여전히 싱글 스레드)

## Q2) OS 레벨에서 멀티 스레드 vs 멀티 프로세싱

### 내 답변
멀티 프로세싱은 모른다고 답변

### 모범 답변
**멀티 스레드 (Multi-threading)**
- 하나의 프로세스 내에서 여러 스레드가 실행
- **메모리 공유**: Heap, Code, Data 영역을 공유 → 스레드 간 통신 빠름
- **단점**: 공유 자원 접근 시 Race Condition 발생 가능 → Mutex, Lock 필요
- **Context Switch 비용**: 프로세스보다 가벼움 (스택만 교체)

**멀티 프로세싱 (Multi-processing)**
- 여러 프로세스가 독립적으로 실행
- **메모리 격리**: 각 프로세스는 독립된 메모리 공간 사용 → 한 프로세스 크래시가 다른 프로세스에 영향 없음
- **IPC 필요**: 프로세스 간 통신은 Pipe, Socket, Shared Memory 등 별도 메커니즘 필요
- **Context Switch 비용**: 스레드보다 무거움 (전체 메모리 컨텍스트 교체)

| 항목 | 멀티 스레드 | 멀티 프로세싱 |
|------|------------|--------------|
| 메모리 | 공유 | 독립 |
| 통신 속도 | 빠름 | 느림 (IPC) |
| 안정성 | 낮음 (공유 오염 위험) | 높음 (격리) |
| Context Switch | 가벼움 | 무거움 |
| 예시 | Java Thread, Go goroutine | Nginx worker process, fork() |

**Java 관점 추가:**
- JVM은 멀티 스레드 기반
- OS 스레드와 1:1 매핑 (Java 21 Virtual Thread 이전)
- Java 21 Virtual Thread: 경량 스레드로 OS 스레드를 점유하지 않고 블로킹 처리 가능

## Q3) N+1 문제 해결 방법

### 내 답변
- Fetch Join 사용
- 연관관계를 ID 기반으로 변경
- @EntityGraph 사용

### 모범 답변
내 답변에서 핵심은 맞음. 더 구조적으로 정리:

**N+1 문제란?**
- 1개의 쿼리로 N개의 엔티티를 조회한 후, 각 엔티티의 연관관계를 로딩하기 위해 추가로 N개의 쿼리가 발생하는 현상

**해결 방법:**

1. **Fetch Join** (JPQL)
   ```java
   @Query("SELECT o FROM Order o JOIN FETCH o.items")
   List<Order> findAllWithItems();
   ```
   - 연관 엔티티를 한 번의 쿼리로 함께 조회
   - 단점: 페이징과 함께 사용 시 `HibernateJpaDialect` 경고 발생 (컬렉션 Fetch Join + 페이징은 메모리에서 처리)

2. **@EntityGraph**
   ```java
   @EntityGraph(attributePaths = {"items"})
   List<Order> findAll();
   ```
   - JPQL 없이 선언적으로 Fetch Join과 동일한 효과

3. **Batch Size (@BatchSize / default_batch_fetch_size)**
   ```yaml
   spring.jpa.properties.hibernate.default_batch_fetch_size: 100
   ```
   - N개의 쿼리를 `IN` 절로 묶어 처리 → N+1을 1+1로 줄임
   - 페이징과 함께 사용 가능 ✅

4. **연관관계 제거 + ID 기반 조회**
   - 연관관계를 끊고 별도 Repository에서 ID로 조회
   - 도메인 간 결합 낮추는 DDD 관점에서 유효한 방법

**선택 기준:**
- 단순 조회 + 페이징 없음 → Fetch Join / @EntityGraph
- 페이징 필요 → Batch Size
- 도메인 분리 필요 → ID 기반 조회

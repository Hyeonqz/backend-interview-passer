## Q2) RESTful API 원칙이란?

### 내 답변
몰라서 대답 못함

### 모범 답변
REST(Representational State Transfer)의 6가지 제약 조건:

| 원칙 | 설명 |
|------|------|
| Stateless | 서버는 클라이언트 상태를 저장하지 않음. 요청마다 필요한 정보가 모두 포함되어야 함 |
| Client-Server | UI와 데이터 저장을 분리. 각자 독립적으로 진화 가능 |
| Cacheable | 응답은 캐시 가능 여부를 명시해야 함 (Cache-Control 헤더 등) |
| Uniform Interface | 자원(Resource)은 URI로 식별, HTTP 메서드(GET/POST/PUT/DELETE)로 행위 표현 |
| Layered System | 클라이언트는 서버가 단일인지 중간 계층(LB, Gateway)이 있는지 알 필요 없음 |
| Code on Demand (optional) | 서버가 실행 코드(JS 등)를 전달할 수 있음 |

실무 포인트:
- URI에 동사 사용 금지 (`/getUser` ❌ → `/users/{id}` ✅)
- 응답에 적절한 HTTP 상태 코드 사용 (200, 201, 400, 404, 409 등)
- HATEOAS까지 언급하면 플러스 (응답에 관련 링크 포함)

## Q3) gRPC, REST-API가 있는데 왜 Kafka Request & Reply 패턴으로 동기식 처리를 했는지?

### 내 답변
Kafka Request & Reply 패턴의 이점을 설명했지만 명확하게 대답하지 못함

### 모범 답변
이 질문의 핵심은 "REST/gRPC가 있는데 굳이 Kafka를 동기식으로 쓴 이유가 있는가?" — 즉, 설계 결정에 대한 근거를 묻는 것.

**Kafka R&R을 선택할 수 있는 근거:**
1. **기존 Kafka 인프라 활용** — 이미 비동기 처리를 위해 Kafka가 구축된 상황에서, 추가 인프라(gRPC 서버) 없이 요청-응답 패턴 구현 가능
2. **감사 로그 / 이벤트 소싱 필요** — 요청과 응답 모두 Kafka 토픽에 남아 재현·추적이 용이
3. **서비스 간 느슨한 결합** — 응답 서비스가 다운되어도 메시지는 Kafka에 보존됨

**솔직하게 인정해야 할 트레이드오프:**
- 순수 동기 통신 목적이라면 REST/gRPC가 지연 시간, 구현 복잡도 측면에서 더 적합
- Kafka R&R은 reply-to 토픽 관리, correlation-id 매칭 등 추가 복잡도 수반
- "해당 프로젝트에서는 이벤트 추적과 느슨한 결합이 더 중요한 요구사항이었기 때문에 선택했고, 순수 응답 속도만 본다면 gRPC가 맞다"고 설명하는 게 좋음

> 면접관 의도: 설계 결정을 맹목적으로 따른 게 아니라 트레이드오프를 이해하고 있는지 검증

## Q4) 기억에 남는 장애 상황 — 원인, 해결, 재발방지

### 내 답변
결제사 장애로 1시간 동안 결제 불가 → Prometheus + Grafana 기반 메트릭 수집 및 알람 체계 수립

### 모범 답변 포인트 (STAR 구조로 다듬기)
- **Situation**: 결제사 측 장애로 외부 API 호출 실패, 1시간 동안 결제 전체 차단
- **Task**: 장애를 빠르게 인지하고 사용자 영향 최소화
- **Action**:
  - 즉시 결제사 상태 페이지 및 담당자 연락
  - 사용자 향 에러 메시지 "일시적 결제 오류" 안내 처리
  - 장애 중 실패한 요청 로그 수집 및 재처리 대기
- **Result + 재발방지**:
  - Prometheus로 외부 결제 API 응답시간 / 실패율 메트릭 수집
  - Grafana 알람: 실패율 N% 초과 시 슬랙 알림
  - 결제사 장애 시 Circuit Breaker 적용 검토 (Resilience4j)

추가 언급하면 좋은 것: 장애 후 Post-mortem(사후 분석) 작성 여부

## Q5) 타 부서와 의사소통 중 생긴 문제와 해결 방법

### 내 답변
영업팀의 일정 조율 미흡으로 트러블 발생 → 내가 결정 사항을 정리해 통보하는 방식으로 진행

### 모범 답변 포인트
현재 답변은 "내가 결정하고 통보"라는 부분이 독단적으로 들릴 수 있음. 아래처럼 재구성 권장:

- **상황**: 영업팀과 개발 일정 협의 과정에서 회의가 반복 지연, 의사결정 공백 발생
- **내 판단**: 무기한 대기보다 명확한 기준 제시가 팀 전체에 이롭다고 판단
- **행동**: 협의 사항을 정리한 문서를 먼저 작성 → 검토 요청 → 기한 내 피드백 없을 시 그대로 진행하는 방식으로 전환
- **결과**: 일정 가시성 확보, 이후 영업팀에서도 해당 방식을 선호하게 됨

핵심: "통보"가 아니라 "명확한 프로세스 제안"으로 프레이밍

## Q6) Kafka DLQ 설계 — 어떤 용도로 사용했는지

### 내 답변
재처리 용도가 아닌 모니터링 및 운영팀 전달 용도로 사용

### 모범 답변
현재 답변은 솔직하고 나쁘지 않음. 보완 포인트:

**DLQ 일반 설계 원칙:**
1. **재시도 전략**: 컨슈머에서 N회 재시도 후 실패 시 DLQ로 이동 (RetryTemplate / `@RetryableTopic`)
2. **DLQ 활용 방안**:
   - 운영팀 알림 (현재 사용 방식 ✅)
   - 수동 또는 자동 재처리 파이프라인
   - 장애 원인 분석용 메시지 보존
3. **재처리 메커니즘**: DLQ 토픽 메시지를 원본 토픽으로 replay하는 별도 Consumer/도구 필요

**추가 언급 포인트:**
- DLQ 메시지에 실패 원인(exception 정보, 재시도 횟수)을 헤더에 포함시키면 운영 효율 증가
- "현재는 모니터링 용도였지만, 재처리 파이프라인 구축의 필요성을 느꼈고 개선 방향을 검토 중이다"고 마무리하면 좋음

## Q7) 모놀리식 → 멀티 모듈 + 헥사고날 아키텍처 도입 이유

### 내 답변
- 공통 코드 비대화, 계층 간 의존 혼재, 빌드 시간 증가, 단일 장애점 문제
- 멀티 모듈로 분리하면서 레이어/코드 구성 방법이 가장 고민됐음
- 헥사고날은 현재 상황에서 체감 이점이 크진 않지만, 테스트 용이성 및 요구사항 변경 시 코드 변경 용이함

### 모범 답변 포인트
답변 자체는 잘 했음. 더 탄탄하게 만들 포인트:

**멀티 모듈 도입 근거 정리:**
- 공통 모듈 비대화 → 변경 시 전체 빌드, 불필요한 의존성 전파
- 도메인 경계 불명확 → 어느 레이어에서나 다른 레이어 직접 호출

**헥사고날 체감 이점 구체화:**
- 외부 시스템(DB, Kafka, 외부 API) 교체 시 Port/Adapter만 수정
- 도메인 로직을 인프라 없이 단위 테스트 가능 → 테스트 속도 향상
- "아직 대규모 교체가 없어서 체감이 적지만 테스트 구조가 명확해졌다"는 현재 답변이 오히려 솔직하고 좋은 평가 받을 수 있음

**추가 질문 대비:**
- "멀티 모듈에서 공통 모듈 의존성은 어떻게 관리했는가?" → core 모듈 분리, 순환 의존 방지
- "헥사고날에서 트랜잭션 경계는 어디에 두었는가?" → Application Service 레이어

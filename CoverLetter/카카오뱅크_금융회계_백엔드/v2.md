## Q1. 카카오뱅크에 입사를 희망하게 된 구체적인 이유는 무엇이며, 입사 후 목표를 기술해 주세요.

> [글자수: 약 820 / 1000]

핀테크 스타트업에서 QR 결제 중계 시스템을 처음부터 설계하고 운영하면서, 금융 데이터에서 단 한 건의 누락이 고객 신뢰를 어떻게 흔드는지 직접 경험했습니다.
POS↔서버 간 결제 상태 불일치로 월 10건의 CS가 반복되던 문제를 Final API 패턴으로 해결하고, SFTP 파일 전송 실패 시 유실 감지조차 불가능하던 구조를 Transactional Outbox 패턴으로 바꿔 파일 유실률 100%를 0%까지 낮추면서, '금융 데이터의 정합성이 곧 고객의 신뢰'라는 확신이 생겼습니다.

카카오뱅크 금융회계 시스템을 선택한 이유는 그 정합성 요구가 가장 엄격한 곳이기 때문입니다.
수천만 거래의 원장 정합성을 실시간으로 보장하고, 결산 단계에서 단 한 건의 이벤트 누락도 허용하지 않는 구조는 제가 실전에서 깊이 다뤄온 문제와 직접 맞닿아 있습니다.
기존 은행과 달리 이 정합성을 외부 솔루션에 의존하지 않고 엔지니어가 직접 설계하고 데이터로 책임지는 문화가 카카오뱅크를 선택한 결정적인 이유입니다.

입사 후 2년 안에 금융회계 결산 파이프라인의 이벤트 정합성을 높이는 작업에 직접 기여하고 싶습니다.
Outbox 패턴과 DLQ 재처리 체계를 상용 환경에서 검증한 경험을 바탕으로, 결산 누락·중복 없는 안정적인 파이프라인 설계에 즉시 투입될 수 있다고 판단합니다.

## Q2. 같지만 다른 은행인 카카오뱅크, 본인의 차별화 된 역량 및 기술을 근거로 지원한 직무에 적합한 이유를 기술해 주세요.

> [글자수: 약 950 / 1000]

카카오뱅크 금융회계 시스템에서 "같지만 다른"이 의미하는 것은 이벤트 하나의 유실도 허용하지 않는 정합성을 엔지니어가 직접 기술로 책임진다는 것이라 생각합니다. 저는 그 책임을 상용 서비스에서 이미 실천해왔습니다.

금융 시스템에서 이벤트 유실은 곧 데이터 불일치입니다.
SFTP 파일 전송 실패 시 사용자는 성공 응답을 받지만 백엔드는 유실을 감지조차 못하던 구조를, Transactional Outbox 패턴으로 전송 이벤트를 DB에 영속화하고 DLQ 기반 재처리 체계를 설계해 파일 유실률 100%→0%를 달성했습니다.
결산 파이프라인에서 이벤트 누락이 허용되지 않는 요구사항은 이 경험과 정확히 맞닿아 있습니다.

분산 환경의 상태 정합성도 직접 해결했습니다.
POS 결제 완료 후 서버 상태 미동기화 문제는 Final API 패턴으로 결제 최종 상태를 단방향 확정하고 재시도 멱등성을 보장해 CS 월 10건→0~1건(95%↓)을 달성했습니다.
Kafka Request-Reply 환경에서는 CorrelationId 기반 응답 격리 설계로 인스턴스 간 타임아웃을 100% 제거했고, TCP 채널 소유권 불일치 문제는 Kafka produce/consume 분리와 Redis 기반 소유 서버 판별로 API↔TCP 서버를 완전히 디커플링했습니다.

성능 개선은 항상 측정에서 시작했습니다.
Pinpoint APM으로 병목을 특정한 뒤 Redis Cache-Aside 패턴과 트랜잭션 경계 재설계로 API 응답시간 200ms→30ms(85%↓)를 달성했습니다. 감이 아닌 지표로 증명되는 개선이 제 일하는 방식입니다.

규제 준수도 코드로 직접 내재화했습니다.
PCI DSS 컴플라이언스 달성 프로젝트에서 HashiCorp Vault 기반 시크릿 관리, Keycloak 2FA, Kafka 기반 감사 로그 추적 체계를 직접 구축했으며, 보안 취약점 High 3건·Medium 5건을 해결하며 금융당국 선불 라이센스 취득에 기여했습니다.

"단 한 건의 거래도 누락·중복 없이" — 이 원칙을 실전 상용 서비스에서 검증한 엔지니어로서, 카카오뱅크 금융회계 시스템에서 즉시 기여할 수 있다고 자신합니다.

## Q1. 카카오뱅크에 입사를 희망하게 된 구체적인 이유는 무엇이며, 입사 후 목표를 기술해 주세요.

> [글자수: 약 800 / 1000]

핀테크 스타트업에서 QR 결제 중계 시스템을 처음부터 설계하고 운영하면서, 금융 도메인에서 단 한 건의 거래 오류가 실제 고객의 신뢰를 어떻게 흔드는지 몸소 경험했습니다. POS와 간편결제사 간 거래 상태 불일치로 월 10건의 CS가 반복되던 상황에서, SSE·Webhook·Final API 방식을 직접 시장 조사하고 사내 PT를 통해 팀을 설득하여 구현까지 주도했습니다. CS가 95% 감소하는 결과를 보며, '금융 데이터의 정합성이 곧 고객의 신뢰'라는 신념이 생겼고, 더 큰 규모의 금융 시스템에서 그 신념을 실현하고 싶다는 목표로 자연스럽게 이어졌습니다. 일 50건이던 거래량이 5,000건으로 100배 성장하는 과정에서 분산 시스템의 확장성과 정합성을 함께 고민해온 경험이, 지금 저를 이 지원으로 이끌었습니다.

카카오뱅크를 선택한 이유는 기술이 서비스의 본질인 조직이기 때문입니다. 기존 은행과 달리 수천만 고객의 금융 데이터를 실시간으로 정확하게 처리하는 기술 인프라 위에 모든 서비스가 세워지며, 엔지니어의 결정이 직접 고객 경험으로 이어집니다. 특히 금융회계 시스템은 단순한 데이터 처리를 넘어, 분산 이벤트 환경에서의 트랜잭션 정합성과 대규모 데이터의 일관성이 핵심인 도메인입니다. 이는 제가 실전에서 가장 깊이 다룬 문제와 직접 맞닿아, 지금 당장 기여할 수 있는 역량을 가진 곳이라 확신했습니다.

입사 후에는 정산 파이프라인의 이벤트 정합성 보장에 즉시 기여하고 싶습니다. Transactional Outbox와 Kafka 기반 분산 메시징으로 유실·중복 없는 금융 이벤트 처리 구조를 설계한 경험이 그대로 적용될 수 있습니다. 

실시간 정산 API의 병목을 APM으로 특정하고 캐싱·쿼리 최적화로 개선하는 작업, PCI DSS 기준의 감사 로그·시크릿 관리 체계를 코드에 내재화하는 작업까지, 상용 서비스에서 직접 검증한 역량들입니다. 

AWS Bedrock 기반 LLM 배치로 운영팀 반복 업무를 95% 자동화하고 AI 활용법을 직접 교육한 경험처럼, 코드 너머 AI를 실무 도구로 내재화하는 방향에도 함께 기여하고 싶습니다. 카카오뱅크의 금융 기술 기준을 코드와 AI로 함께 높여가는 엔지니어가 되는 것이 목표입니다.

## Q2. 같지만 다른 은행인 카카오뱅크, 본인의 차별화 된 역량 및 기술을 근거로 지원한 직무에 적합한 이유를 기술해 주세요.

> [글자수: 약 910 / 1000]

카카오뱅크를 "같지만 다른 은행"이라고 표현할 때, 저는 그 차이가 어디서 오는지 압니다. 수천만 거래를 외부 솔루션에 기대지 않고 내부에서 직접 설계하고 책임지는 것, 그리고 그 정합성을 엔지니어가 데이터로 증명하는 문화입니다.

저는 핀테크 스타트업에서 QR 결제 중계 시스템을 처음부터 구축하며 비슷한 긴장감을 몸으로 경험했습니다. POS와 간편결제사 사이에서 거래 상태가 엇갈리면 CS가 직결되고, 잘못된 정합성은 신뢰의 손상 및 회사의 손실로 이어집니다.

Transactional Outbox 패턴으로 파일 유실률을 100%에서 0%로 끌어내렸고, Final API 설계로 상태 불일치를 95% 줄였을 때, 숫자보다 CS 문의가 사라지던 순간의 안도감이 더 선명하게 남아 있습니다.

성능 개선 역시 감이 아닌 측정에서 시작했습니다. Pinpoint APM으로 병목을 특정한 뒤, Redis Cache-Aside와 트랜잭션 경계 재설계로 API 응답시간을 200ms에서 30ms로 줄였습니다. 

Kafka Request-Reply 아키텍처에서는 인스턴스별 Reply 토픽과 CorrelationId 설계로 분산 환경의 타임아웃 문제를 완전히 제거했습니다.

보안·컴플라이언스 영역도 직접 경험했습니다. PCI DSS 준비 프로젝트에서 HashiCorp Vault 기반 시크릿 관리와 Keycloak 2FA, Kafka 기반 감사 로그 체계를 구축했고, 금융당국 선불 라이센스 취득에도 핵심 역할로 참여해 규제 요건을 실제 코드로 옮기는 경험을 쌓았습니다.

카카오뱅크가 FDS에 AI를 도입해 이상거래 탐지를 고도화하는 방향처럼, 저 역시 AI를 실무 자동화 도구로 직접 다룬 경험이 있습니다. AWS Bedrock(Claude) 기반 LLM 배치를 설계·구현해 운영팀의 수동 업무를 월 100건에서 5건으로 줄였고, 현재는 운영팀을 대상으로 AI 활용법을 교육하며 비개발 직군과 AI가 함께 작동하는 협업 구조를 만들어가고 있습니다.

금융회계 시스템에서 "같지만 다른"이 의미하는 것은 기술로 정합성을 직접 책임진다는 것이라 생각합니다. 그 책임을 실전 상용 서비스에서 검증한 엔지니어로서, 즉시 기여할 수 있다고 자신합니다.

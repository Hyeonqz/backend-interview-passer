## Q1) 영속성 컨텍스트란 무엇이며 Redis 캐시와 어떻게 다른가
> 영속성 컨텍스트가 무엇인지, 그리고 Redis 캐시와 어떻게 다른지 설명해주세요.

## A1) 내 답변
영속성 컨텍스트는 JPA 메커니즘에서 트랜잭션이 있는 상황에서 캐시 역할을 한다.
RDB까지 가서 엔티티를 조회하는게 아닌 영속성 컨텍스트에서 엔티티를 로드해두고 조회하는 방식이다.
Redis 캐시는 완전히 Save 해두고 사용하는 거지만, 영속성 컨텍스트는 트랜잭션이 종료되면 사라진다.

## 모범 답변

영속성 컨텍스트(Persistence Context)는 JPA가 엔티티를 관리하는 논리적 저장소다.
EntityManager가 이를 관리하며, 기본적으로 트랜잭션과 생명주기를 같이한다.

**핵심 기능 5가지**
1. **1차 캐시**: 동일 트랜잭션 내 같은 PK 조회 시 DB 재접근 없이 캐시에서 반환
2. **동일성 보장**: 같은 PK로 조회하면 항상 `==` 동일 인스턴스 반환
3. **변경 감지(Dirty Checking)**: 트랜잭션 커밋 시 스냅샷과 비교해 변경된 엔티티 자동 UPDATE
4. **지연 쓰기(Write-Behind)**: INSERT/UPDATE를 즉시 실행하지 않고 flush 시점에 일괄 실행
5. **지연 로딩(Lazy Loading)**: 연관 엔티티를 실제 접근 시점에 쿼리

```java
// 1차 캐시 동작 예시
@Transactional
public void example(String txId) {
    Transaction tx1 = txRepository.findById(txId); // DB 쿼리 실행
    Transaction tx2 = txRepository.findById(txId); // 1차 캐시 반환, DB 쿼리 없음
    System.out.println(tx1 == tx2); // true - 동일성 보장
}
```

**Redis 캐시 vs 영속성 컨텍스트 1차 캐시 비교**

| 구분 | 영속성 컨텍스트 1차 캐시 | Redis 캐시 |
|------|--------------------------|------------|
| 범위 | 단일 트랜잭션 (기본) | 프로세스/서버 경계를 넘어 공유 |
| 생명주기 | 트랜잭션 종료 시 소멸 | TTL 기반, 명시적 삭제 전까지 유지 |
| 저장 대상 | JPA 엔티티 객체 | 직렬화된 값 (String, JSON 등) |
| 목적 | 동일 트랜잭션 내 중복 쿼리 방지 + Dirty Checking | 서버 재시작/스케일 아웃 환경에서 DB 부하 분산 |
| 분산 환경 | 인스턴스별 독립적 | 모든 인스턴스 공유 |

**참고**: OSIV(Open Session In View) 패턴을 활성화하면 영속성 컨텍스트 범위가
트랜잭션이 아닌 HTTP 요청 전체로 확장된다. Spring Boot는 기본값이 OSIV 활성화 상태이므로
운영 환경에서는 명시적으로 비활성화(`spring.jpa.open-in-view=false`)를 검토해야 한다.

## Q2) 조회 트랜잭션에서 엔티티 변경 시 발생하는 문제와 방어 방법
> `@Transactional`이 걸린 조회 메서드 안에서 엔티티 필드를 실수로 변경하면 어떤 일이 발생하나요? 조회 전용 API에서 방어하는 방법은 무엇인가요?

## A2) 내 답변
엔티티 필드 변경 시 영속성 컨텍스트 로드된 내용과 달라져 데이터 불일치 문제가 발생한다.
방어 방법은 모르겠다.

## 모범 답변

**오개념 정정**: "데이터 불일치"가 아니라, 변경 감지(Dirty Checking)가 정상 작동해서
**의도치 않은 UPDATE 쿼리가 실제 DB에 자동 실행**되는 것이 핵심 문제다.

```java
// 문제 상황
@Transactional
public TransactionStatus getStatus(String txId) {
    Transaction tx = txRepository.findById(txId).orElseThrow();
    tx.setStatus(TransactionStatus.CANCELLED); // 실수로 변경
    return tx.getStatus();
    // 메서드 종료 → 트랜잭션 커밋 → Dirty Checking 실행
    // → 스냅샷과 비교 후 변경 감지 → UPDATE 쿼리 자동 실행
    // → DB의 거래 상태가 CANCELLED로 바뀌어버림
}
```

**방어 방법 3가지**

방법 1. `@Transactional(readOnly = true)` (가장 권장)
- Dirty Checking 자체를 수행하지 않음 (스냅샷 미생성 → 메모리 절약)
- flush 모드가 NEVER로 설정되어 실수로 변경해도 UPDATE 쿼리 발생 안 함
- 일부 DB 드라이버/리플리케이션 환경에서 Read Replica로 자동 라우팅 가능

```java
@Transactional(readOnly = true)
public TransactionStatus getStatus(String txId) {
    Transaction tx = txRepository.findById(txId).orElseThrow();
    tx.setStatus(TransactionStatus.CANCELLED); // 변경해도 UPDATE 쿼리 발생 안 함
    return tx.getStatus();
}
```

방법 2. DTO Projection으로 직접 조회 (QueryDSL/JPQL)
- 엔티티 자체를 영속성 컨텍스트에 올리지 않으므로 Dirty Checking 대상 자체가 없음

```java
@Transactional(readOnly = true)
public TransactionStatusDto getStatus(String txId) {
    return queryFactory
        .select(Projections.constructor(TransactionStatusDto.class,
            transaction.id,
            transaction.status))
        .from(transaction)
        .where(transaction.id.eq(txId))
        .fetchOne();
}
```

방법 3. `@Transactional` 제거
- 단순 캐시 조회 → DB 조회 흐름에서 트랜잭션이 불필요하면 제거
- 단, JPA Lazy Loading이 필요한 경우 `LazyInitializationException` 발생 주의

```java
public TransactionStatus getStatus(String txId) {
    // Cache-Aside 패턴에서 Redis hit이면 트랜잭션 없이 바로 반환
    return redisTemplate.opsForValue().get("tx:" + txId);
}
```

# Context_Switching_최적_스레드수

## Q1)
> 실시간 데이터 처리 서버에서 Context Switching 이 성능에 미치는 영향이 중요해요. <br>
> Context Switching이 무엇인지 설명하고, 스레드가 많아질수록 오히려 성능이 떨어지는 이유를 설명해주세요. <b>
> 그리고 실시간 처리 서버에서 최적의 스레드 수를 어떻게 결정하시겠어요?


## A1) 내 답변
```text
컨텍스트 스위칭은 OS 에서 스레드에게 작업을 할당할 때 일어나는 비용입니다.
스레드가 많아질수록 컨텍스트 스위칭도 잦아져서, 성능이 저하가 될 수 있습니다.
CPU 코어는 한정적이기 때문이죠

스프링부틑 기준 최적의 스레드는 OS 레벨에서 cpu 코어와  목표 tps 를 측정하여 
계산하는 방식이 있습니다.
```


## 모범 답변
```text
최적 스레드 수 공식:

CPU Bound 작업 (계산 집약적):
최적 스레드 수 = CPU 코어 수 + 1
→ 코어보다 많으면 Context Switching만 증가
→ 예: 암호화, 데이터 압축

I/O Bound 작업 (대기 시간 많음):
최적 스레드 수 = CPU 코어 수 × (1 + 대기시간/처리시간)
→ I/O 대기 중 다른 스레드가 CPU 사용 가능
→ 예: DB 조회, 외부 API 호출, Kafka 메시지 처리

실시간 데이터 처리 = I/O Bound
→ Kafka 메시지 대기, DB 저장 대기 시간 많음
→ 코어 수보다 많은 스레드가 유효

Spring Boot Tomcat 기본값:
server.tomcat.threads.max=200 (기본값)
→ 실제 서버 스펙에 맞게 튜닝 필요

실무 접근법:
1. 부하 테스트(nGrinder, JMeter)로 목표 TPS 측정
2. 스레드 수 조정하며 최적값 탐색
3. CPU 사용률 70~80% 유지가 목표
   (100%면 Context Switching 과부하)
```

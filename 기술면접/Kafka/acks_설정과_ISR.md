## Q1) 
> 이력서를 보면 QR 결제 중계 시스템을 처음부터 설계하고 팀을 이끌어오셨는데요. <br>
> 만약 지금 iVX에서 차량 운행 데이터를 수집하고 운영 대시보드에 실시간으로 표시하는 시스템을 새로 설계해야 한다면, 어떤 아키텍처로 접근하시겠어요? <br>
> 현재 보유한 기술 스택 기준으로 큰 그림을 그려주세요.

## A1)
```text
실시간 데이터를 안정적으로 받아 운영 대시보드 표시하기 위해서는 기술 자체는 Java or Kotlin 프레임워크는 SpringBoot 를 사용하여 개발할 것 같습니다
초당 수십만건의 데이터를 안정적으로 처리하기 위해서 Kafka 아키텍쳐를 사용할 것 같고  내부 아키텍쳐로는 이벤트 드리븐 아키텍쳐를 채택할 것 같습니다. 
데이터가 조금이라도 유실되면 안돼는 점을 감안했을 때 Kafka 에서는 조금 성능은 느리지만 안정적으로 처리할 수 있는 장치 acks-=all, 가 있기에 실시간 데이터를 안정적으로 처리하기에 적합하다고 판단하였습니다.
```


## 모범 답변
```java
[수집 레이어]
차량 → REST or MQTT → Gateway → Kafka Topic
                                (파티션 키: vehicleId)

[처리 레이어]
Kafka Consumer Group → 이벤트 파싱 → DB 저장 (시계열 DB or MySQL)
                    → 이상 감지 → Alert 발행

[조회 레이어]
운영 대시보드 → Redis 캐시 → DB
             → SSE or WebSocket (실시간 Push)

[안정성 장치]
- acks=all + min.insync.replicas=2 → 메시지 유실 방지
- DLQ → 처리 실패 메시지 별도 관리
- Consumer Lag 모니터링 → 처리 지연 감지
```


## Q2) 
> 방금 "Kafka에서 안정적으로 처리할 수 있는 장치가 있다"고 하셨는데요. <br>
> 구체적으로 Kafka Producer 레벨에서 메시지 유실을 방지하기 위한 설정이 무엇인지, 그리고 그 설정이 성능에 어떤 트레이드오프를 만드는지 설명해주세요.

## A2) 
```text
producer 레벨에서는 acks=all 설정을 통해 토픽 설정 시에 설정한 ISR 만큼
리더 브로커가 아닌 팔로워 브로커에 복제가 되므로 안정적이라고 생각했습니다.

위 설정이 안정적이긴 하지만 성능적으로 봤을 떄 유실확률이 제로이고, 그 만큼 카프카 메시지가 팔로워 브로커에 쓰기 까지 조금더 시간이 걸릴것 같습니다.

카프카가 제로카피를 이용하여 디스크에 쓰는 방식이 최적화되어있다고 하지만, 수십만건의 데이터를 처리할때는
분명 병목이 있을 수 있다는 생각이 듭니다.
```


## 모범 답변
```text
acks=all + ISR 복제 메커니즘을 정확히 연결해서 설명했어요. 단순히 설정값만 외운 게 아니라 리더-팔로워 복제 흐름까지 이해하고 있다는 게 드러났어요.
Zero Copy 개념까지 언급한 부분도 인상적이에요. Kafka 내부 최적화를 알고 있다는 신호예요.

한 가지 추가하면 좋은 포인트
병목 가능성을 언급했는데, 여기서 면접관은 "그럼 어떻게 해결하겠어요?"를 바로 물어볼 거예요. 미리 한 문장을 붙여두면 완성도가 올라가요.

"병목이 발생할 경우 파티션 수를 늘리고 Consumer Group을 수평 확장하거나, 배치 처리(linger.ms, batch.size 튜닝)로 처리량을 높이는 방향으로 접근하겠습니다."
```

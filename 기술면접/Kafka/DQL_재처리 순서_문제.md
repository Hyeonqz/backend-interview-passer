## Q1) 
> 수동 커밋 + DLQ 전략을 사용할 때, 같은 메시지가 DLQ에 중복으로 쌓이는 상황을 방지하려면 어떻게 설계해야 하나요? <br>
> 그리고 DLQ에 쌓인 메시지를 재처리할 때 원본 토픽의 순서와 무관하게 처리해도 되는지, 결제 또는 차량 이벤트 도메인 특성상 순서가 중요한 경우 어떻게 접근하시겠어요?

## A1) 
```text
실시간 데이터를 순차적으로 처리해야 한다고 하면 문제가 발생할 것 같습니다.
데이터 중복 처리를 하지 않기 위해서는 멱등키 방법을 채택할 것 같습니다.

또는 DLQ 를 처리할 때는 Kafka 에서 딱 한번 처리를 보장하는 트랜잭션 프로듀서를 사용할 것 같습니다.

위 데이터가 어떤 데이터인지는 모르겠습니다만, 토픽 순서 무관하게 처리? 이 부분은 잘 모르겠습니다.
```


## 모범 답안
```text
DLQ 재처리 시 순서 문제:

원본 토픽:
[offset 1] 결제요청 → 정상처리
[offset 2] 결제취소 → 실패 → DLQ 이동
[offset 3] 정산요청 → 정상처리

DLQ에서 offset 2를 나중에 재처리하면
이미 offset 3(정산)이 완료된 상태
→ 취소가 정산 이후에 처리되는 순서 역전 발생

해결 방법:
- DLQ 재처리 시 이벤트 timestamp 또는 sequence 검증 후 처리
- 순서가 critical한 도메인은 DLQ 자동 재처리보다 수동 검토 후 처리
- 결제처럼 순서가 중요한 경우: DLQ → 알람 → 수동 판단 후 처리
```
